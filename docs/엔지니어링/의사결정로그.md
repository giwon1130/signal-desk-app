# 의사결정 로그 (ADR)

> **누가 봄**: 모든 개발자
> **언제 봄**: 큰 결정 직전 (전례 검색) / 결정 직후 (기록)
> **규칙**: 큰 결정만. 작은 코드 변경은 git log 로 충분.

## 형식

```
## ADR-XXX — YYYY-MM-DD
**결정**: 한 줄
**대안**: 검토했던 다른 옵션들
**선택 이유**: 왜 이걸 골랐나
**영향**: 다른 코드/설정에 미치는 영향
**되돌리기 비용**: 낮음 / 중간 / 높음
**관련**: (있으면 PR/이슈 링크)
```

시간 역순 정렬 (최신이 위).

## ADR-008 — 2026-04-26
**결정**: IP 기반 Rate Limit (Bucket4j in-memory) 도입
**대안**: Redis 기반 분산 limit, Cloudflare WAF, nginx limit_req
**선택 이유**:
- 단일 인스턴스 단계엔 메모리로 충분
- Bucket4j 는 Spring 통합 단순
- Redis 추가는 비용 + 복잡도 ↑
**영향**: scale-out 시 IP limit 이 인스턴스마다 따로 — Redis 마이그 필요
**되돌리기 비용**: 낮음 (`RateLimitFilter.kt` 삭제)
**관련**: PR `feat/rate-limiting`

## ADR-007 — 2026-04-26
**결정**: Web 호스팅 `serve@14` (Node) → `nginx:alpine`
**대안**: Caddy, Vercel Static, Railway Static 사이트
**선택 이유**:
- 메모리 70MB → 10MB
- gzip + immutable cache 자체 지원
- Railway 와의 호환성 검증됨
**영향**: `railway.toml` 의 `startCommand` 제거 필요 (nginx 이미지엔 serve 없음)
**되돌리기 비용**: 낮음 (Dockerfile 한 파일)
**관련**: PR `chore/cost-savings-nginx`

## ADR-006 — 2026-04-26
**결정**: WS 가격 폴링에 시장 OPEN 가드 추가
**대안**: 5초 → 30초로 그냥 늘리기 / 사용자별 구독 시간 제한
**선택 이유**:
- 휴장 시간엔 외부 호출 자체가 무의미
- KR/US 별도 체크 → 한쪽만 OPEN 일 때 그쪽만 fetch
**영향**: 외부 호출 ~50% ↓, Naver 차단 위험 ↓
**되돌리기 비용**: 낮음
**관련**: PR `chore/cost-savings-railway`

## ADR-005 — 2026-04-25
**결정**: 가격 표시에 통화 prefix (`₩` / `$`) 강제
**대안**: 시장 컬럼만 따로 두기, 단위 명시 없이 숫자만
**선택 이유**:
- KR `₩119,000` 옆에 US `$412.43` 같이 떠도 단위 혼동 방지
- formatPrice(value, market) 한 함수로 통일
**영향**: 모든 가격 표시 코드 일괄 갱신 (10+ 파일)
**되돌리기 비용**: 중간

## ADR-004 — 2026-04-25
**결정**: KR 휴장일을 `MarketSessionService` 에 하드코딩 (자동 계산 X)
**대안**: 외부 캘린더 API, 라이브러리 (`korean-calendar` 류)
**선택 이유**:
- 음력 + 대체휴일 + 임시공휴일 자동 계산은 신뢰성 낮음
- KRX 공식 캘린더 매년 12월 공시 → 수동 갱신이 안전
**영향**: 매년 12월 코드 갱신 필요 (`KR_HOLIDAYS_BY_DATE` map)
**되돌리기 비용**: 낮음
**관련**: PR `feat/kr-holiday-calendar`

## ADR-003 — 2026-04-24
**결정**: AI 탭을 "로그 뷰어" 에서 "플레이북 + 성적표" 로 재정의
**대안**: 로그 뷰어 그대로 유지 (단순)
**선택 이유**:
- 사용자가 로그 뷰어 안 씀 (피드백)
- 플레이북 = 행동 / 성적표 = 검증 → 매일 두 번 들어올 이유
**영향**: 모바일 + 웹 두 화면 동시 재작성
**되돌리기 비용**: 높음 (다시 짜야 됨)

## ADR-002 — 2026-04 (초기)
**결정**: 단일 코드베이스 (Expo) — iOS + Android + Web
**대안**: RN (mobile only), Next.js (web only) 분리
**선택 이유**:
- 1인 개발 단계 — 코드 중복 비용 큼
- Expo SDK 54 의 RN-Web 성숙도 충분
- `Platform.OS === 'web'` 분기로 데스크톱 전용 UX 가능
**영향**: 웹 전용 기능은 `src/web/` 하위, 공유는 `src/` 직속
**되돌리기 비용**: 높음 (분리 시 mobile/web 따로 짜야)

## ADR-001 — 2026-04 (초기)
**결정**: Spring Boot Kotlin 백엔드 + Railway 배포
**대안**: Node.js (Express/Fastify), Go, Python FastAPI, Vercel Serverless
**선택 이유**:
- Kotlin/JVM 친숙도
- Spring 의 풍부한 어노테이션 (인증/CORS/스케줄링)
- Railway 의 Docker 빌드 편의 + Postgres 통합
**영향**: 배포 비용 always-on (cold start 회피)
**되돌리기 비용**: 매우 높음
