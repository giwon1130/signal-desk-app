## 핵심 원칙

(stub)
1. **한 코드베이스, 두 플랫폼**: Expo 가 iOS/Android/Web 빌드. `Platform.OS === 'web'` 분기로 웹 전용 셸 적용.
2. **얇은 백엔드**: Spring Boot 가 외부 데이터 수집·캐싱·sentiment 분석·사용자 데이터만. 트레이딩 실행 없음.
3. **외부 API 깨져도 살아있음**: 모든 외부 호출 `runCatching` + fallback.
4. **휴장 시 외부 호출 안 함**: 5초 WS 폴링도 시장 OPEN 만.
5. **클라이언트 캐시 안 함**: useMarketSnapshot 매번 fetch (서버 60s/3m TTL 이 보호).

## 데이터 흐름

(stub)
1. **시장 데이터**:
   `KRX/Naver/FRED/News → MarketOverviewService (캐시 60s/3m) → /api/v1/market/* → App`
2. **사용자 데이터**:
   `App → /api/v1/workspace/* → WorkspaceController → JDBC → Postgres`
3. **실시간 가격**:
   `App ↔ WS → PriceTickerHandler (5s polling, OPEN 시간만) → Naver`
4. **인증**:
   `App → /auth/login or /auth/oauth/google → AuthService → JWT → AsyncStorage`

## 핵심 결정 (왜)

(stub) "왜 이렇게 만들었나" 의 SoT.
- **왜 캐시 60s**: KRX/Naver 외부 호출 비용 절감 + 실시간 5s WS 가 별도 채워줌
- **왜 시장별 5s 폴링**: 거래소 직접 연결 권한 없음. Naver 단일 큰 채널 비용 절감.
- **왜 휴장 가드**: Railway outbound + Naver 차단 위험 (cost.md 참고)
- **왜 nginx 웹**: Node serve 메모리 70MB → 10MB
- **왜 AGENTS.md 분리**: Claude / 자동화는 사람용 docs 와 톤 다름
