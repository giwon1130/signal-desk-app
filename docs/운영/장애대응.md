# 장애 대응 (Runbook)

> **누가 봄**: 운영자 / 온콜
> **언제 봄**: 장애 발생 / 사용자 신고 / 모니터링 알람

## 장애 분류 + 대응

### 🔴 API 다운 (`/health` 200 아님)

(stub)
- 증상: 앱 새로고침 시 "서버에 연결할 수 없어요" 에러
- 1차 확인: `railway logs --service signal-desk-api`
- 흔한 원인:
  - JVM OOM (`java.lang.OutOfMemoryError`) → 인스턴스 재시작 + 메모리 limit 확인
  - DB 연결 끊김 (`Connection refused`) → Postgres 서비스 상태 확인
  - 외부 API timeout 폭증 → fallback 동작 확인
- 빠른 조치: Railway 대시보드에서 redeploy

### 🟡 외부 데이터 없음 (KRX/Naver/FRED 죽음)

(stub)
- 증상: 시장 요약 비어있음, 가격 0
- 백엔드는 `runCatching` + fallback 으로 살아있음 — UI 만 빈 카드
- 1차 확인: API 로그에서 `... fetch failed` warn 빈도
- 조치: 외부 API 회복 대기. 우리가 할 거 없음.

### 🟡 Postgres 다운

(stub)
- 증상: 워크스페이스 CRUD 만 실패 (`/api/v1/workspace/*` 500)
- 시장 요약은 메모리 캐시 + 외부 데이터로 생존
- 1차 확인: Railway Postgres 서비스 상태
- 조치: Railway 대시보드에서 Postgres restart

### 🟡 CORS 차단

(stub)
- 증상: 웹에서 `Access-Control-Allow-Origin` 에러
- 원인: `SIGNAL_DESK_CORS_ALLOWED_ORIGIN_PATTERNS` 에 도메인 누락
- 조치: 환경변수 갱신 → 자동 재시작

### 🟡 인증 모두 실패 (`401` / `400`)

(stub)
- JWT secret 변경 직후라면 — 모든 사용자 토큰 무효화. 재로그인 필요.
- audience 화이트리스트 변경 직후라면 — Google ID 누락 확인

### 🔴 Rate limit 오작동 (정상 사용자도 429)

(stub)
- 원인: 한 IP 에 여러 사용자 (corporate NAT 등)
- 조치: 한도 상향 (`RateLimitFilter.kt` 의 `MARKET_RULE.capacity`)

## 로그 보는 법

```bash
# API
railway logs --service signal-desk-api
railway logs --service signal-desk-api --build

# 특정 키워드 필터 (예: rate-limit)
railway logs --service signal-desk-api 2>&1 | grep "rate-limit"

# 운영 로그 마커
# - [SECURITY] = 보안 경고 (JWT/audience)
# - rate-limit = rate limit 차단
# - "fetch failed" = 외부 API 실패
```
